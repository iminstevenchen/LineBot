# 育兒導航全攻略 LINE Bot — 我的負責部分說明

> 本文件說明本專案中我負責實作的部分：後端架構、資料庫設計、對話流程、RAG 整合、LLM 生成邏輯。
> 供隊友製作 PPT 使用。

---

## 一、整體架構概覽

```
使用者
  │  LINE App
  ▼
LINE Platform（雲端）
  │  Webhook POST /callback          ← 被動（使用者發訊息才觸發）
  ▼
Flask Server（本地 / Render 雲端）
  ├─ onboarding.py      ← 新手引導流程
  ├─ main.py            ← 訊息路由、非同步 Q&A
  ├─ db.py              ← 資料庫 CRUD
  ├─ rag.py             ← 知識庫查詢（隊友的 AnythingLLM）
  ├─ llm.py             ← Groq LLM 生成回覆
  └─ push_scheduler.py  ← 定時主動推播（每天 09:00 執行）
  │
  ▼
Supabase PostgreSQL（雲端資料庫）
```

**為什麼這樣設計？**
LINE Bot 的 Webhook 要求在 5 秒內回應，但 RAG + LLM 通常需要 5–15 秒。所以流程分兩段：
1. 收到訊息 → 立即 Reply「思考中，請稍候 🤔」（在 5 秒內完成）
2. RAG + LLM 在背景 Thread 執行 → 完成後用 **Push Message** 把真正答案送給使用者

## 五、新手引導（Onboarding）狀態機

### 流程圖

```
加入好友（FollowEvent）
        │
        ▼
waiting_parent_name  →  waiting_phone_number
        │
        ▼
  waiting_city（兩層：先選地區 → 再選縣市）
        │
        ▼
waiting_parental_employment（就業狀況）
        │
        ▼
waiting_child_name  →  waiting_child_birthday（DatetimePicker）
        │
        ▼
waiting_child_gender  →  waiting_birth_order
        │
        ▼
waiting_special_status（16 種特殊身分，可多選）
        │
        ▼
      done  ←─── 正式問答開始
```

### 設計決策

**為什麼用「state machine」？**
LINE Bot 是無狀態的（每次 Webhook 都是獨立請求）。必須把「用戶現在回答第幾個問題」存在資料庫，下一次才能接續流程。狀態機讓每個步驟的轉換規則清晰且容易測試。

**兩層縣市選擇（地區 → 縣市）**
LINE Quick Reply 最多 13 個按鈕。台灣有 22 個縣市，所以先選北/中/南/東/離島，再顯示該地區的縣市卡片。

**DatetimePicker 選生日**
讓使用者用 LINE 原生的日期選擇器（不用手動輸入文字），避免格式錯誤，也更方便。選擇後觸發 PostbackEvent，由 `handle_postback()` 接收。

**特殊身分（16 種）**
收集早產兒、低收入戶、單親家庭等，讓 LLM 可以提供更精準的補助資訊（例如：早產兒有特別補助方案）。

### 修改資料 / 多寶寶管理

使用者完成引導後，仍可：
- 輸入「修改資料」→ 選擇要改家長資料或哪位寶寶的資料
- 輸入「新增寶寶」→ 進入第二個寶寶的引導流程
- 輸入「切換寶寶」→ 用卡片輪播選擇活躍的寶寶

這些修改流程也是用 `editing_*` 狀態機實作，和初始引導共用同一個 `process()` 函式。

## 六、Q&A 問答流程（主流程）

### 完整流程（按順序，不可改動順序）

```python
# 1. 先撈歷史（必須在 save_message 之前，否則當前訊息混入自己的歷史）
history = db.get_recent_history(user_id, limit=20)

# 2. 儲存當前訊息
db.save_message(user_id, "user", text)

# 3. 立即回覆「思考中」（Reply Token 在 5 秒內失效）
_send_messages(reply_token, [TextMessage("思考中，請稍候... 🤔")])

# 4. 背景執行 RAG + LLM
threading.Thread(target=_process_qa_async, ...).start()
```

### 背景執行（`_process_qa_async`）

```python
rag_query  = llm.build_rag_query(text, context, history)  # 組查詢字串
rag_chunks = rag.query_rag(rag_query)                      # 查知識庫
reply      = llm.generate_reply(text, context, rag_chunks, history)  # LLM 生成
db.save_message(user_id, "assistant", reply)               # 儲存回覆
db.trim_chat_history(user_id, keep=30)                     # 清理舊紀錄
_push_text(user_id, reply)                                 # Push 給使用者
```

**為什麼要先 Reply 再 Push？**
LINE Webhook 的 `reply_token` 只有 5 秒有效，但 RAG + LLM 要 5–15 秒。所以先用 Reply 送「思考中」佔住 Token，再用不需要 Token 的 Push Message 送真正答案。

### 3 歲以上提醒

如果寶寶生日計算出已超過 36 個月，回覆前會自動加一則警示訊息：「育兒小幫手目前主要服務 0～3 歲寶寶…」這是考量到服務定位，避免提供不適用的資訊。

## 七、RAG 知識庫整合

### 架構

```
main.py
  │ 呼叫 rag.query_rag(rag_query)
  ▼
rag.py
  ├─ 有設定 RAG_API_URL → POST 到隊友的 AnythingLLM
  └─ 未設定 / 連線失敗  → fallback 到 rag_mock.py（本機模擬）
```

### RAG Query 的組法（`llm.build_rag_query`）

```
[背景：使用者：小明；寶寶生日：2024-03-15，今天：2026-06-28；用戶說：寶寶一直哭；助理說：可能是腸絞痛...] 我的寶寶最近睡眠很差怎麼辦？
```

把使用者名字、寶寶生日、最近兩輪對話上下文一起帶進去，讓知識庫知道「這是幾個月大的寶寶在問什麼」，查出更精準的片段。

### Fallback 設計

如果隊友的 AnythingLLM 連不上（Cloudflare Tunnel 重啟後 URL 會變），自動切換到 `rag_mock.py`（7 個硬編碼知識塊，關鍵字比對）。系統不會因為 RAG 掛掉而整個停止運作。

## 八、LLM 生成邏輯

### 模型

**Groq `llama-3.3-70b-versatile`**
- 選用原因：Groq 推論速度極快（Token/s 遠高於一般 API），且免費額度充足，適合回應速度有要求的 LINE Bot。

### System Prompt 設計原則

System Prompt 包含四大塊：

1. **角色定位**：溫暖的育兒助理，透過 LINE 服務台灣新手爸媽
2. **回覆原則**：優先用知識庫資料；誠實承認不知道；推薦 1922 諮詢專線
3. **安全邊界**（不可違反）：
   - 不做具體診斷
   - 不建議調整藥物
   - 緊急狀況（呼吸困難、抽搐）→ 說「請撥 119」
   - 疑似兒虐/家暴 → 保護專線 113
   - 自傷風險 → 安心專線 1925
4. **格式規定**：不使用 Markdown（LINE 會原文顯示 `**` `#`），改用 `•` 條列、emoji、【括號】

### 完整 Prompt 結構（每次問答）

```
[System Prompt]
---
[User Prompt]
--- 使用者資料 ---
使用者暱稱：小明
今天日期：2026-06-28
寶寶生日/預產期：2025-01-15

--- 近期對話記錄 ---
用戶：寶寶一直哭
助理：可能是腸絞痛，建議...

--- 知識庫內容 ---
【知識片段 1】
...來自 RAG 的相關育兒知識...

--- 使用者最新問題 ---
睡眠不好怎麼辦？

請根據以上資訊，以溫暖友善的語氣回覆：
```

### 歷史對話壓縮

當歷史超過 `_SUMMARY_THRESHOLD`（20 筆）時：
- 較早的部分 → 呼叫 LLM 壓縮成 100 字內的摘要
- 最近 4 筆 → 保持原文

目前設定是「20 筆才摘要」，等同「永不摘要」（最多撈 20 筆），這是為了省掉額外的 Groq API 呼叫、加快回應速度。

### 重試機制

`generate_reply` 失敗最多重試 3 次，每次間隔 2 秒。所有嘗試都失敗才回傳備用錯誤訊息。

## 十、部署架構

系統有兩種部署方式，差異在於 Flask App 跑在哪裡。

---

### 方式一：全部跑在振東的虛擬機（Cloudflare Tunnel）

```
LINE Platform
      │  Webhook POST /callback
      ▼
Cloudflare Tunnel（自動建立公開 HTTPS，不需固定 IP）
https://saying-daisy-drum-constant.trycloudflare.com
      │
      ▼
振東的虛擬機（本機）
  ├─ Flask App（main.py，port 8000）
  ├─ AnythingLLM（RAG 知識庫）
  └─ 呼叫 Groq API、Supabase（外部雲端服務）
```

- Flask App 和 AnythingLLM **都在同一台 VM 上**跑，不需要 Render
- Cloudflare Tunnel 把 VM 上的 Flask 暴露成公開 HTTPS，填入 LINE Developers Console 當 Webhook URL
- **缺點**：Tunnel URL 每次重啟都會變，需要去 LINE Console 更新 Webhook URL

**Flask 啟動方式（VM 上）：**
不依賴 PaaS，在 VM 上直接執行 gunicorn：
```bash
gunicorn main:app --bind 0.0.0.0:8000 --workers 2 --timeout 120
```
`PORT` 固定 8000，Cloudflare Tunnel 對應到這個 port。

> 備注：AnythingLLM（RAG 知識庫）在同一台 VM 上的架設與啟動方式，請見振東的說明文件 `docs/linebot_dev_plan.md`。

---

### 方式二：Flask App 部署到 Render（ngrok + UptimeRobot）

```
LINE Platform
      │  Webhook POST /callback
      ▼
Render（雲端 PaaS，固定 HTTPS URL）
  ├─ Flask App（main.py）
  └─ 呼叫 Groq API、Supabase

UptimeRobot ──每 5 分鐘 ping──▶ Render
（防止 Render 免費方案閒置自動休眠）

Flask App 呼叫 RAG：
  https://headboard-shorthand-unharmed.ngrok-free.dev
      │  ngrok 轉發
      ▼
  本機 / VM 上的 AnythingLLM
```

- Flask App 在 Render 上，LINE Webhook URL 固定不變
- AnythingLLM 仍在本地跑，透過 ngrok 暴露給 Render 上的 Flask 呼叫
- UptimeRobot 定時 ping Render，避免免費方案閒置後休眠
- **優點**：Webhook URL 永久固定，不用每次更新 LINE Console

---

### 兩種方式比較

| | 方式一（VM + Cloudflare） | 方式二（Render + ngrok）|
|---|---|---|
| Flask App 跑在哪 | 振東的虛擬機 | Render 雲端 |
| AnythingLLM 跑在哪 | 振東的虛擬機 | 本機 / VM |
| LINE Webhook URL | 每次重啟後要重貼 | 固定不變 |
| 需要額外服務 | 無 | ngrok + UptimeRobot |
| VM 關機後 Bot 停擺 | 是 | 否（Render 持續運行）|

## 十一、主動推播系統（push_scheduler.py）

前面的 Q&A 流程都是「被動」的——使用者問，Bot 才答。推播系統讓 Bot 可以在正確的時機**主動**提醒家長。

### 三種推播內容

| 類型 | 內容來源 | 推播時機 |
|---|---|---|
| 疫苗 + 兒童健檢 | 台灣 CDC 接種時程（硬編碼）+ 衛福部健檢時程 | 每天 09:00 檢查，里程碑**當天**推一次 |
| 發展里程碑 | 硬編碼里程碑說明（1–36 個月，共 20 個） | 寶寶**滿幾個月的當天**推一次 |
| 政策申請截止提醒 | 隊友的 RAG 知識庫（policy-pipeline 索引的政策文件） | **每週二** 09:00（理由見下方） |

### 避免衝突的設計

**問題：** 疫苗、健檢、里程碑、政策提醒可能在同一天全部觸發，導致使用者一次收到 4 則訊息。

**解法：**

1. **疫苗 + 健檢合併成一則**：同月齡常常同時打針又要做健檢（例如 6 個月：打三針 + 第 4 次健檢），統一整合成一則「健康提醒」訊息，不拆開傳送。

2. **政策提醒移到週二**：隊友的政策爬蟲在週一凌晨執行，若我們在週一 09:00 就推，RAG 索引可能尚未更新完；移到週二可保證推出去的資料是最新的。

3. **里程碑精確到日**：用 `birth_date + N 個月` 算出精確的滿月日，只在那一天推，不會每天都推。

### 推播流程

```
每天 09:00（台灣時間）
  │
  ├─ 從 DB 撈出所有 onboarding_state='done' 且有 birth_date 的用戶
  │
  └─ 對每位用戶：
       ① 計算疫苗 + 健檢到期日 → 若命中 days_until == 0 → 合併成一則推播
       ② 計算今天是否為寶寶的整月紀念日 → 是 → 推里程碑訊息
       ③ 若今天是週二 且 RAG_API_URL 已設定 →
            查詢 RAG → Groq 整理 → 若有截止資訊 → 推政策提醒
```

### 政策提醒如何整合 policy-pipeline（含來源連結）

```
push_scheduler.py
  │  query = "縣市 月齡 育兒補助 政策 申請期限 截止日"
  ▼
rag.query_rag_with_sources() → POST 到隊友的 AnythingLLM
  │
  ├─ answer：政策文字內容
  └─ sources：[{ title, url }, ...]  ← AnythingLLM 附帶的原始文件 metadata
  │
  ▼
llm._chat()（Groq）整理成 80 字內的純文字摘要
  │  若 Groq 回覆「無相關截止提醒」→ 本週不推
  ▼
組裝最終訊息：
  摘要內容
  + （若 sources 有 https:// URL）📎 資料來源\n標題\nURL
  │  LINE 自動把 URL 轉成可點擊連結
  ▼
_line_push() → 送出 LINE Push Message
```

隊友的 policy-pipeline 每週一更新文件索引，推播系統每週二讀取，確保推出去的政策資訊永遠是最新一批。

## 十二、為什麼這樣做（設計理念總結）

| 決策 | 原因 |
|---|---|
| 用狀態機做引導 | LINE Webhook 無狀態，需靠 DB 記住「用戶在哪一步」|
| 立即 Reply + 背景 Push | LINE Reply Token 只有 5 秒，RAG+LLM 需要更長時間 |
| children 獨立一張表 | 支援多胎家庭，`active_child_id` 做「目前問哪個寶寶」的指針 |
| RAG fallback 到 mock | 隊友伺服器可能不穩定，確保 Bot 不因知識庫掛掉而停擺 |
| Groq llama-3.3-70b | 推論快、免費額度夠、回應品質佳 |
| 不用 Markdown | LINE 平台不支援，會原文顯示符號，改用 emoji 和括號強調 |
| 3 歲以上警示 | 服務定位是 0–3 歲，超齡提早告知避免資訊誤用 |
| 16 種特殊身分收集 | 台灣各縣市補助因身分不同差異大，收集後可提供精準的補助查詢 |
| 疫苗只推當天 | 只在到期當天推一次，不提前預告，避免騷擾使用者 |
| 疫苗 + 健檢合併一則 | 同月齡常同時觸發，分開傳送體驗差；合併讓訊息更結構化 |
| 政策提醒改到週二 | 爬蟲週一凌晨跑，週二推才能確保 RAG 索引已更新完畢 |
| 里程碑精確到滿月日 | 避免每天推同一個月的里程碑；只在寶寶確切整月紀念日推一次 |
| push_scheduler 獨立不 import main | main.py 啟動 push_scheduler，若反過來會循環 import；push_scheduler 直接建 LINE ApiClient 解決 |
